2. 프로젝트의 빌드결과로 만들어지는 파일의 위치를 변경하는법
  0) 이 작업을 하는 이유
    - 프로젝트들을 빌드하면 각 프로젝트의 파일안에 x64라는 파일이 생기고 그 안에 빌드완료로 인한 결과파일들이 저장됨.
    - 그런데 이렇게 되면 나중에 빌드결과 파일들을 싹다 삭제하고 다시 빌드 하고 싶을때 일일히 프로젝트 파일 찾아 들어가면서 x64 파일을 삭제해야함
      프로젝트 개수가 100개면 100개 파일 일일히 들어가면서 x64 파일을 삭제하는 것임
    - 이런 소모적인 일을 피하기 위해서 솔루션파일 하위 목록에 임의의 파일 만들고 그 파일 안에 모든 프로젝트들의 빌드결과 파일들을 몰아서 저장시킬것임

  1) 프로젝트 이름 우클릭 -> 속성(R) -> 구성 속성 -> 일반 -> 출력 디렉터리, 중간 디렉터리 변경  
    - 출력 디렉터리는 빌드의 최종 결과물에 관한 파일, 중간 디렉터리는 빌드의 중간 결과물에 관한 파일.

    - $(매크로이름) : $()기호는 해당 '매크로이름'에 맞는 파일경로로 알아서 바꿔주는 기능이다.
      예시 - $(SolutionDir) : 솔루션의 파일경로
           - &(Platform) : 운영체제에 처리형식에 맞는 파일이름 (64비트면 x64)
      매크로는 출력 디렉터리 오른쪽에 있는 아래화살표 클릭 -> <편집> -> 매크로(M)>> 에서 다양한 매크로 키워드를 확인할 수 있음.

  2) 출력 디렉터리 경로 - $(SolutionDir)Bin\$(Platform)\$(Configuration)\ 
     중간 디렉터리 경로 - $(SolutionDir)Bin\$(Platform)\$(Configuration)\$(ProjectName)

3. 종속성 설정
  0) 이 작업을 하는 이유
    - 빌드할때 프로젝트 하나씩 일일이 클릭해서 빌드하는 소모적인 일을 하지 않기위해, 한 프로젝트를 빌드하면 연관된 다른 
      프로젝트들도 같이 빌드 시키는 설정이다.

  1) 솔루션 이름 우클릭 -> 속성(R) -> 프로젝트 종속성 -> 프로젝트(R) 에는 빌드하는 프로젝트, 다음에 종속(D)에는 같이 빌드할 프로젝트를 박스눌러서 체크

4. 다중 프로세서 컴파일 설정
  0) 이 작업을 하는 이유
    - 싱글코어로 컴파일 하는거 보다 당연히 멀티코어로 하는게 빌드 속도가 더 빠름. 요즘엔 대부분의 프로세스들이 멀티코어이기 때문에
      이 설정을 해주자

  1) 프로젝트 이름 우클릭 -> 구성 속성 -> C/C++ -> 일반 -> 다중 프로세서 컴파일 -> 예(/MP)
    - 주의 : 프로젝트 내에 아무 파일이 없으면 C/C++ 선택지가 안뜸. 프로젝트에 아무 파일 만들면 C/C++ 목록이 생긴다.
